# 主动联系

使用 cc-connect Timer 获得主动联系用户的机会。区分长期延续的“链式主动关心”和针对具体事情的“一次性临时回访”。

## 链式主动关心

收到包含“根据时间和前面聊的内容”的提示时，把它视为链式主动关心触发。

1. 根据本轮时间 Hook 提供的准确时间、当前会话和最近对话，自主判断是否适合联系用户。
2. 自由决定下一次触发间隔。结合当前时间、聊天节奏以及用户是否可能正在睡觉或忙碌，不使用固定间隔表。
3. 设置且只设置一个下一次 Timer：

```powershell
cc-connect timer add --delay Xm --prompt "根据时间和前面聊的内容判断要不要主动联系对方，要发就正常发，不发就沉默，然后记得要设置下一次timer" --desc "链式主动关心"
```

需要按小时延迟时把 `Xm` 换成 `Xh`。

4. 设置下一次 Timer 后，再输出自然的聊天内容；不适合联系时只输出精确的 `NO_REPLY`。

不要在正文中提及 Timer、触发任务、判断流程或系统机制。普通聊天结束时不要重复创建链式主动关心；只有当前链式 Timer 触发，或者用户明确要求启动、恢复时才创建。

## 创建临时回访

只有当前对话出现一件确实会在稍后产生结果、状态变化或值得自然关心的事情时才创建。不要机械地给每段对话设置回访。

根据事情本身自由决定延迟时间，把必要情境直接写进提示词，不要使用“根据刚刚的对话”这种会失去指向的说法。

```powershell
cc-connect timer add --delay Xm --prompt "临时回访：用户在TIME提到EVENT。先检查当前会话里是否已经有结果；已经有结果就只输出NO_REPLY；还没有结果就自然地关心或询问。不要提及Timer、回访任务或系统机制。这是一次性回访，不要设置下一次timer。" --desc "临时回访：EVENT_SHORT"
```

执行时替换：

- `TIME`：事情发生的时间，只写有助于理解的精度。
- `EVENT`：完整但简短的具体事情。
- `EVENT_SHORT`：方便识别和取消的短名称。
- `Xm`：自然的等待时间，也可以使用 `Xh`。

临时回访触发后：

- 当前会话已经出现结果时，输出精确的 `NO_REPLY`。
- 事情仍未有结果时，像正常聊天一样自然询问。
- 无论是否发送消息，都不要为这次回访继续创建 Timer。
- 链式主动关心与临时回访互相独立，不要把回访描述写成“链式主动关心”。

## 提前结束回访

用户在 Timer 到期前已经主动给出结果，而当前会话中能确定存在对应回访时，使用 `cc-connect timer list` 找到描述为 `临时回访：EVENT_SHORT` 的任务，再用 `cc-connect timer del <id>` 取消。

无法确定对应哪一项时不要误删；保留 Timer，触发后由其中的“已有结果则 NO_REPLY”规则兜底。
